# 2026-05-07 round156 + Codex security review fix

参照: `private_docs/2026-05-07_round156_codex_review_findings.md`

## 即時対応 (本コミット)

### R156-S1 (CRITICAL): `/api/youtube_live/start` URL 検証 bypass

**問題**: Pydantic `StartRequest.url` に `min_length` / `max_length` しか制約が
なく、`probe_hls()` が False を返した DRM フォールバック分岐 (`create_drm_job`)
で生 URL がそのまま `RecordJob.url` に格納されていた。round156 で `https://[::1]/`
や `javascript:alert(1)` を含む 13 種の不正 URL がすべて 200 でジョブ化されることを
動的攻撃で確認。

**修正 (2 段の防御)**:

1. `routers/youtube_live.py::_validate_public_https_url(value)` を追加し
   `StartRequest.url` の `field_validator` として登録。Pydantic 層で:
   - 制御文字 / DEL / 改行 reject
   - scheme=https のみ許可
   - embedded credentials reject
   - hostname 必須、`localhost` 系文字列 reject
   - IP 直指定なら loopback / private / link-local / reserved / multicast / unspecified reject
2. `services/youtube_live_recorder.py::_validate_drm_capture_url(url)` を追加し
   `create_drm_job` 内で再検証 (defense-in-depth)。`_validate_url_for_subprocess` は
   yt-dlp/ffmpeg subprocess の引数として http も許可するが、screen capture 経路は
   Electron `loadURL` に渡るため https のみに絞る。

ローカル単体検証: 13/13 不正 URL が両層で reject、3/3 正常 URL が両層を通過することを確認。

`backend/tests/test_youtube_live_url_validation.py` を回帰テストとして追加 (5 テスト
ファミリ × 計 30+ ケース)。CI で常時回す。

### SEC-001 (HIGH): `is_production_posture` プロパティで本番姿勢ガードを統一

**問題**: docs / benchmark / TrustedHost / 危険ルータの 4 ガードが `PUBLIC_MODE`
単体や `PUBLIC_MODE or HIDE_API_DOCS` の OR 判定にバラバラに分散しており、
`ENVIRONMENT=production` だけで運用しているケース・`SS_PUBLIC_HOSTNAME` 公開ホスト
判定を漏らすケースがあった。

**修正**:
- `Settings.is_production_posture` プロパティを追加。判定基準は
  `PUBLIC_MODE` / `HIDE_API_DOCS` / `HIDE_STACK_TRACES` /
  `ENVIRONMENT=production` / `SS_PUBLIC_HOSTNAME` のいずれかが立っていれば True
- `main.py` の docs URL / benchmark router / 危険ルータ群 / CORS origins /
  TrustedHost の各ガードを `is_production_posture` 一本に統合
- 起動時に `[posture] is_production=...` を log に明示出力 (運用時 sanity check)
- 旧 `_PRODUCTION_INTENT` 変数 (config.py) は `settings.is_production_posture` に
  置換し削除

### bootstrap-status 露出 (round128 / 攻撃集計 #3)

**問題**: `GET /api/auth/bootstrap-status` が無認証で `{"has_admin":..., "bootstrap_configured":...}`
を返し、`app.shuttle-scope.com` 経由でも reconnaissance に使えていた。

**修正**: 本番姿勢時は admin 認証必須。匿名 / 非 admin には `has_admin` のバイナリ
判定だけ返して内部状態を隠す。LAN / Electron 単体運用時は Electron 初回 UX のため
従来通り匿名で許可。

### SEC-005 (MEDIUM): Ray dashboard `0.0.0.0` 既定 → `127.0.0.1`

**問題**: `backend/main.py:262` の Ray auto-start が `--dashboard-host=0.0.0.0` で
LAN 全体に Ray dashboard (port 8265) を露出していた。

**修正**: 既定 `127.0.0.1`。`SS_RAY_DASHBOARD_HOST=0.0.0.0` env を明示設定した時のみ
LAN 公開 (Cloudflare Access / Basic auth 等の追加認証層併用が前提)。

### R156-S7 (NORMAL): warning schema drift

**修正**: `_job_status` は既に `getattr(job, "warning", None)` で defensive に
key 含めていたが、round156 で `schema_ok=False` となった原因は production が古い
build (warning フィールド未追加 RecordJob) で動いていたため。本コミット deploy 後の
新規 job では schema 統一されることを round156 再実行で確認予定。

### Pre-existing: 未宣言 env 変数 5 つを `Settings` に declare

`.env.development` の以下 5 変数が `Settings` 未宣言で `extra_forbidden` 起動拒否を
出していた。本番無影響だが unit test の `import` を阻害していたため declare のみ追加:
- `SS_OTP_TTL_MINUTES`
- `SS_TEMP_PASSWORD_TTL_HOURS`
- `SS_SENDER_MAX_RECORDING_DURATION_MIN`
- `SS_PUBLIC_REQUEST_TTL_HOURS`
- `VITE_SS_BILLING_UI_ENABLED`
