# ShuttleScope バドミントン業界覇権戦略

最終更新: 2026-04-16

## 0. 結論

ShuttleScope が狙うべきポジションは、Dartfish の代替ではない。単なる映像比較ツールでもない。狙うべきは、バドミントン専用の「試合運用・注釈・分析・意思決定・共有・研究」までを一体化したプラットフォームである。

勝ち筋は明確である。

1. バドミントンに特化したタグ設計・試合運用で、汎用映像解析ソフトを上回る
2. ダブルス分析を中核差別化要素として研究から製品へ引き上げる
3. 映像レビューを目的にせず、試合データを資産化し、次の意思決定に使える状態を作る
4. ローカル運用・現場運用・チーム内共有を強くし、導入障壁を下げる
5. 特許を先行取得し、競争相手が後から追随しにくい地形を作る

## 1. 基本認識

### 1.1 何と戦うのか

ShuttleScope が本当に戦う相手は、単一製品ではない。

- Dartfish のような映像解析ソフト
- 手打ちのスコアシート
- Excel 集計
- コーチの記憶と勘
- 試合後に動画を見返すだけの運用
- 分析担当者ごとに分散した属人的メモ

つまり、ShuttleScope が置き換えるべきものは「既存ソフト 1 本」ではなく、現場に散らばった非効率な運用全体である。

### 1.2 勝負を間違えるな

Dartfish の強みは、映像比較表現、商用品質、汎用性、洗練されたレビュー体験にある。

ShuttleScope が正面からそこを模倣すると、劣化コピーにしかならない。しかも Simulcam / Stromotion のような代表機能に寄せるほど、特許・意匠・製品印象の両面で不利になる。

したがって、勝負の軸は以下に固定する。

- バドミントン専用ワークフロー
- ダブルス分析
- 試合中・セット間の意思決定支援
- 半自動注釈とレビュー効率化
- 研究成果を製品へ高速実装できる基盤

## 2. 目指すべき製品定義

ShuttleScope は以下の定義で磨く。

「バドミントン専用の試合運用・注釈・分析・共有・予測・研究統合 OS」

これにより、単なる動画プレイヤーではなく、チームの競技力を継続的に上げるインフラになる。

### 2.1 製品の中核価値

- 試合をラリー・ショット単位で構造化できる
- ダブルスの前衛後衛・打者・役割分担まで扱える
- 現場で破綻しにくい入力フローを持つ
- 試合後に自動で分析・プレイリスト・レポートへ流せる
- 次戦の意思決定やラインナップ選定まで繋げられる
- 研究中の分析ロジックを現場に落とし込める

### 2.2 顧客に売る言葉

売るべき言葉は「動画をきれいに比較できます」ではない。

売るべきは以下である。

- 試合をデータ資産に変える
- ダブルスの見えない構造を可視化する
- 試合中・セット間の判断を速くする
- アナリスト依存の属人運用を減らす
- 選手・コーチ・アナリストが同じ情報基盤を見る

## 3. 市場での勝ち筋

## 3.1 第一の勝ち筋: ダブルス分析

ダブルス分析は明らかにブルーオーシャン寄りである。

シングルスは比較的単純で、既存の映像解析や一般的な統計でもそこそこ戦える。一方、ダブルスは以下の複雑性がある。

- 前衛後衛の役割分担
- 打者以外の位置・動きの価値
- ローテーションの質
- 守備から攻撃への遷移
- 触れていない選手の存在価値
- パートナー相性
- 左右・利き手・陣形の組み合わせ
- 配球の読み合いと介入タイミング

ここは従来のタグ集計だけでは弱い。だから研究余地が大きく、製品差別化もしやすい。

ShuttleScope はここを中核テーマにするべきである。

## 3.2 第二の勝ち筋: 現場運用のしやすさ

どれだけ理論が優れていても、試合現場で回らないなら無価値である。

勝つためには、以下を徹底する。

- 入力ミスが起きにくい UI
- 試合進行に追従できる速度
- 見逃しラリーの後処理のしやすさ
- セット間速報の自動化
- コーチ向け要約の即時生成
- Viewer / LAN / ローカル共有の強さ

研究より先に運用で死ぬ製品は多い。ここを軽視すると覇権はない。

## 3.3 第三の勝ち筋: 半自動化

完全自動化を目指して現場投入が遅れるのは愚かである。今必要なのは完全自動ではなく、半自動で人間を速くすることだ。

狙うべきは以下である。

- 自動候補生成
- 候補の確認キュー
- ワンクリック採用
- 不確実箇所だけ人間が判断
- 修正ログの蓄積
- 将来の再学習資産化

この流れなら、精度が 100% でなくても価値がある。

## 4. 開発優先順位

## 4.1 最優先

### A. ダブルス分析エンジン

やるべきこと:

- 前衛後衛の役割推定
- ローテーションパターン分類
- 攻守遷移の分類
- 介入成功 / 介入不足の抽出
- ペア相性スコア
- ペア別の得点・失点構造の可視化
- 相手ペアへの有効戦術抽出

製品価値:

- 他社が簡単に真似しづらい
- 研究成果がそのまま差別化になる
- バドミントン特化の旗印として強い

### B. 高速レビュー導線

やるべきこと:

- 条件フィルタから即プレイリスト化
- 特定局面の自動抽出
- 失点パターンの即列挙
- セット間で見せるべき 3〜5 本の自動推薦
- コーチ向け短文要約

製品価値:

- 現場での体感価値が高い
- 試合後レビューの速度が上がる
- 「見返すだけ」から「意思決定する」へ変わる

### C. マルチアングル同期

やるべきこと:

- 同一ラリーに複数動画を紐づける
- 手動・半自動オフセット補正
- 横並び再生
- A/B トグル
- 同期ジャンプ
- タグ連動再生

注意:

- 合成比較は避ける
- 残像系の代表表現を安易に真似しない

製品価値:

- 現場導入で強い
- 特許回避しつつ比較体験を強化できる

## 4.2 第二優先

### D. セット間・試合中支援

やるべきこと:

- 直近ラリー傾向の自動要約
- 相手の偏り検出
- 自軍の崩れ検出
- 次セットの注目ポイント提示
- コーチ向け一言カード

製品価値:

- 試合後分析との差別化
- 単なる記録ソフトから戦術支援ソフトへ進化

### E. データ資産化と同期

やるべきこと:

- パッケージ化
- JSON / CSV / 独自形式の入出力
- チーム内標準スキーマ
- 学習用データ抽出
- 匿名化共有

製品価値:

- 組織導入しやすくなる
- データが継続資産になる

## 4.3 第三優先

### F. 研究カード群の製品化

やるべきこと:

- EPV 的指標の競技文脈最適化
- 状態遷移モデル
- 条件分析の信頼度表示
- 小標本への保守的挙動
- 研究向け画面と現場向け画面の分離

製品価値:

- 研究成果を現場へ落とせる
- ただし過度に難解だとユーザー価値が下がる

## 5. 特許戦略

重要: 以下は事業・製品戦略上の観点であり、法的な特許性判断は弁理士・弁護士と連携して進めること。

## 5.1 基本方針

狙うべきは、単なる一般機能の特許ではない。勝つための特許は、以下のどちらかである。

1. バドミントン特有のデータ構造・処理フロー
2. 現場運用と分析結果を結ぶ独自ワークフロー

つまり、「動画を再生する」「線を引く」のような一般論では弱い。取るなら、競技固有の解釈・処理・提示・運用の組み合わせである。

## 5.2 有望な特許テーマ

### P1. ダブルス役割推定・ローテーション解析

候補:

- 前衛後衛の役割状態推定方法
- ラリー中の役割遷移モデル
- ダブルスの崩れ検知方法
- ペアの介入不足箇所抽出方法
- ダブルス守備から攻撃への遷移評価方法

理由:

- 競技固有性が高い
- 既存一般ソフトが弱い
- 研究成果を権利化しやすい

### P2. 半自動注釈ワークフロー

候補:

- 自動候補提示と人間修正を統合した注釈確定フロー
- 候補信頼度に応じた UI 切替
- 試合進行を止めにくい確認キュー方式
- 修正履歴を再学習資産へ変換する手法

理由:

- 現場での運用差が出る
- 完全自動でないこと自体が強みになる

### P3. セット間・試合中意思決定支援

候補:

- 直近ラリー列からコーチ向け要点を自動生成する方法
- ダブルス戦術修正候補の提示方法
- 相手の偏り・崩れを検知して優先表示する方法
- 試合中に閲覧可能な軽量戦術カード生成

理由:

- 映像解析ソフトから一段上の領域
- 実務価値が高い

### P4. ラリー単位マルチアングル同期

候補:

- ラリー ID を基準とした複数映像の同期管理
- タグイベント基準での多視点同期ジャンプ
- 合成でなく比較導線に最適化した多視点レビュー方式

理由:

- 実装価値が高い
- 合成比較を避けつつ守りを作れる

### P5. バドミントン特化のデータモデル

候補:

- ラリー・ショット・打者・位置・役割・結果を統合したデータ構造
- ダブルスの非打者行動を含む評価スキーマ
- パートナー相性・陣形・局面を横断する集計方法

理由:

- 製品の中核資産を守れる
- 他社が後追いしにくい

## 5.3 特許で避けるべき罠

- 一般的すぎる機能を出願する
- 先行技術調査を甘く見る
- UI の見た目だけを真似しつつ、権利侵害を軽視する
- 実装できない夢機能を大量出願する
- 研究成果がまとまる前に曖昧な出願を乱発する

重要なのは数だけではない。守るべき中核テーマを押さえることだ。

## 5.4 実務上の特許運用

- 毎月、出願候補レビュー会を実施する
- 研究・開発・現場ヒアリングをまとめる
- 発明メモを日付付きで残す
- 実装前でも手法が固まったら仮出しレベルで整理する
- 公開前に出願判断を挟む

## 6. Dartfish と差別化すべき点

## 6.1 やるべき差別化

- バドミントン特化のデータ構造
- ダブルス分析
- 半自動注釈
- セット間意思決定支援
- 試合前予測・ラインナップ支援
- 現場運用と研究の接続

## 6.2 真似しない方がいい領域

- 映像合成比較を主役にすること
- 残像表現をそのまま再現すること
- 汎用スポーツ向け UI を広く持ちすぎること
- 派手だが現場価値の薄い演出中心の進化

## 7. 3段階ロードマップ

## Phase 1: 勝てる土台を固める

目的:

- バドミントン専用ツールとして現場導入で勝つ

やること:

- Annotator の安定化
- 高速レビュー導線
- セット間速報
- Viewer / LAN 共有の強化
- 半自動候補提示の運用化
- データスキーマの固定

成功条件:

- 1 チームが継続利用できる
- 試合後のレビュー時間が明確に短縮される
- コーチが動画を見るだけより有益だと感じる

## Phase 2: ダブルス分析で抜く

目的:

- 他社が持たない競技固有価値を作る

やること:

- ダブルス役割推定
- ローテーション分析
- ペア相性評価
- 介入不足検出
- 守備崩れ検知
- 戦術修正カード
- 主要発明の特許出願

成功条件:

- ダブルス分析が売り文句になる
- 「これがあるから ShuttleScope を使う」と言われる

## Phase 3: 業界標準化

目的:

- バドミントン分析基盤の標準になる

やること:

- チーム間での標準スキーマ普及
- パッケージ流通
- 研究協業
- 指導現場・大学・実業団への展開
- 外部との連携 API 整備
- 継続的な出願・論文化・導入事例づくり

成功条件:

- ShuttleScope 形式のデータが事実上の標準になる
- 分析担当者が最初に触る前提ソフトになる

## 8. 事業上の優先顧客

優先順位は以下。

1. 実業団・大学の上位チーム
2. ダブルス重視のチーム
3. 分析担当がいるが専用基盤を持たない組織
4. 研究と現場を繋ぎたい大学研究室・競技団体

理由:

- 高度な価値が伝わりやすい
- 継続利用されやすい
- 改良フィードバックの質が高い

## 9. KPI

技術 KPI ではなく、競争力 KPI を見る。

- 1 試合あたりの注釈時間
- 試合後レビュー開始までの時間
- コーチが実際に見た自動抽出クリップ数
- セット間要約の利用率
- ダブルス分析機能の継続使用率
- 1 チームあたりの蓄積ラリー数
- 再利用された過去試合データ数
- 製品由来で改善した戦術判断の事例数
- 出願件数ではなく中核テーマの成立件数

## 10. 最後に

覇権を取るには、機能数ではなく、勝つ軸の明確化が必要である。

ShuttleScope が進むべき道は明確である。

- 映像比較ソフトの模倣をやめる
- バドミントン専用の試合運用基盤になる
- ダブルス分析を最大の武器にする
- 半自動化で現場価値を出す
- 意思決定支援まで踏み込む
- 特許を中核テーマに集中させる

この方向で進むなら、後発ではない。むしろ、まだ地形が決まっていない側にいる。

だからこそ、いま必要なのは横に広げることではない。勝てる核心を深く掘ることである。
