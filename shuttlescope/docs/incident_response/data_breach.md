# Data Breach Runbook

DB ファイルやバックアップ ZIP が外部に漏洩した可能性がある場合の対応。

## 影響範囲の判定

### 漏洩物 = 平文 DB ファイル

**最悪ケース** (Phase A 適用前):
- 全選手の体重・体調・氏名・所属が平文取得される
- アノテーションデータ全件
- JWT (revoked_tokens に入っていなければ偽造可)

**Phase A 適用後**:
- 暗号化フィールド (`Condition.injury_notes`, `general_comment`) は SS_FIELD_ENCRYPTION_KEY がなければ復号不可
- 平文のまま漏洩するもの:
  - 数値 (weight_kg, hooper_*, rpe 等)
  - 選手氏名 (Player.name)
  - 試合データ
  - 動画ファイルパス (video_local_path)

### 漏洩物 = バックアップ ZIP

**Phase A 適用後**: AES-256 暗号化されているため SS_BACKUP_PASSPHRASE がなければ復号不可。
ZIP のみ漏洩 + パスフレーズ未漏洩なら**実害なし**。
両方漏洩なら全 DB 平文 + 上記の暗号化フィールドのみ守られた状態。

## 即時アクション (1 時間以内)

1. **影響範囲確定**: どのファイルが、いつ、誰に漏洩したか確認
2. **鍵ローテ**: SS_FIELD_ENCRYPTION_KEY と SS_BACKUP_PASSPHRASE を必ず両方ローテ
3. **JWT 一斉失効**: SECRET_KEY もローテ → 全セッション再ログイン
4. **video_token 一斉再発行**: 動画パスが漏洩していれば再生 URL を全て切り替え
5. **法令対応開始**: 個人情報保護法 / GDPR 等の通報義務確認

## 顧客通知

要配慮個人情報 (体重・体調・怪我情報) が含まれるため、漏洩確定時は速やかな通知が必要。

通知文テンプレート:

```
件名: ShuttleScope データ漏洩のお知らせとお詫び

平素より ShuttleScope をご利用いただきありがとうございます。
このたび、当システム内のデータベースファイルが外部に漏洩した可能性が
判明したため、状況をご報告いたします。

【漏洩した可能性のある情報】
- 選手氏名・所属
- 試合アノテーションデータ
- (Phase A 適用前の場合: 体重・体調メモ等を含む)

【対応状況】
- 本日 HH:MM、暗号化鍵を全てローテーションし、過去の漏洩データの
  解読を不可能にしました
- 全ユーザーの認証トークンを失効させ、再ログインを必須としました
- 動画再生 URL を全て新しいものに切り替えました
...
```

## 事後対応

- ポストモーテム作成
- ファイルシステムレベルの保護強化検討 (Windows ACL、BitLocker)
- 監査ログの改ざん検知強化 (Phase C 残課題)
